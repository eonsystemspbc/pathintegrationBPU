# Optic-Lobe Optic-Flow Benchmark

This benchmark tests whether the fly optic-lobe connectome provides a useful
recurrent initialization for a visual ego-motion task.

The task is inspired by connectome-constrained fly visual-system modelling,
including Lappalainen et al., Nature 2024, where optic-lobe connectome structure
was used to constrain recurrent networks trained on visual motion/optic-flow
objectives.

## Task

Each training example is a short movie rendered onto a hexagonal sensor lattice.

The input sequence is generated by:

1. Procedurally creating a panoramic texture.
2. Sampling it through a fly-like hex lattice.
3. Applying known ego-motion:
   - yaw rotation,
   - forward translation,
   - lateral translation.
4. Rendering the transformed panorama through acceptance-angle blur.
5. Applying a difficulty-controlled contrast/noise/sample-size ladder.

The model receives a tensor:

```text
[batch, timesteps, hex_lattice_samples]
```

and predicts:

```text
[yaw_rate, forward_velocity, lateral_velocity]
```

at every timestep.

## Models

All models are size matched:

- same number of recurrent neurons,
- same number of trainable recurrent edge parameters,
- same input layer shape,
- same readout shape,
- same optimizer/training schedule.

The compared models are:

### `optic_lobe_seeded`

The recurrent graph starts from the prepared FlyWire optic-lobe adjacency:

```text
support = optic-lobe connectome edges
weights = scaled optic-lobe synapse weights
```

The recurrent edge values are trainable.

### `random_weight_topology`

The model preserves:

```text
same neuron count
same edge count
same optic-lobe recurrent support/topology
```

but replaces the optic-lobe synaptic weights with Gaussian random edge weights
matched to the average absolute scale of the prepared adjacency.

This asks whether the optic-lobe graph support alone is helpful when the actual
synaptic weight organization is removed.

### `shuffled_topology`

The model preserves:

```text
same neuron count
same edge count
same recurrent weight multiset
```

but randomizes the recurrent support/topology.

This asks whether the specific optic-lobe wiring arrangement matters.

### `random_sparse`

The model preserves:

```text
same neuron count
same edge count
same trainable parameter count
```

but uses random sparse support and Gaussian recurrent weights.

This asks whether any size-matched sparse recurrent network can solve the task.

## AWS Run

From the repo root on the AWS machine:

```bash
cd /home/ec2-user/pathintegrationBPU/pathintegrationBPU
git pull origin main
source /home/ec2-user/pathintegrationBPU/.venv/bin/activate

OUT=/home/ec2-user/pathintegrationBPU/pathintegrationBPU/outputs/flywire_optic_lobe_flow_medium_3seed
mkdir -p "$OUT"

python scripts/run_optic_flow_benchmark.py \
  --mode all \
  --output-dir "$OUT" \
  --cache-dir "$OUT" \
  --flywire-download-dir /home/ec2-user/pathintegrationBPU/flywire_cache \
  --device cuda \
  --difficulty medium \
  --models optic_lobe_seeded random_weight_topology shuffled_topology random_sparse \
  --seeds 0 1 2 \
  --epochs 30 \
  --patience 8 \
  --batch-size 64 \
  --log-every-seconds 30 \
  2>&1 | tee "$OUT/flywire_optic_lobe_flow_medium_3seed.log"
```

For a quick AWS smoke run before launching the full optic lobe:

## Outputs

The run writes:

- `neurons.csv`
- `roi_counts.csv`
- `connections.csv`
- `pool_assignments.csv`
- `adjacency_unsigned.npz`
- `adjacency_signed.npz` if transmitter-sign coverage supports it
- `graph_metadata.json`
- `metrics_by_seed.csv`
- `metrics_summary.csv`
- `loss_history.csv`
- `optic_flow_loss.png`
- `optic_flow_rmse.png`
- `optic_flow_report.md`
- `run_config.json`

The console log includes explicit `download-*`, `prepare-*`, `matrix-*`,
`model-build-*`, `data-gen-*`, and `progress` events, so AWS runs show where
time is spent during graph preparation, synthetic stimulus generation, and
training.

## Multi-GPU Strict-Prior Sweep

After the optic-lobe adjacency has been prepared, use the multi-GPU launcher to
run model/seed jobs in parallel. The strict-prior version freezes recurrent
edge values and trains only input/readout parameters, which tests whether the
optic-lobe recurrent graph itself is useful rather than merely an editable
sparse mask.

```bash
cd ~/pathintegrationBPU
git pull
source .venv/bin/activate

MATRIX=outputs/flywire_optic_lobe_flow_medium_3seed/adjacency_unsigned.npz
OUT=/mnt/fast/outputs/flywire_optic_lobe_flow_medium_frozen_5seed
mkdir -p "$OUT"

python scripts/run_multi_gpu_associative_sweep.py \
  --benchmark optic_flow \
  --output-dir "$OUT" \
  --gpus 0 1 \
  --status-seconds 15 \
  --models optic_lobe_seeded random_weight_topology shuffled_topology random_sparse \
  --seeds 0 1 2 3 4 \
  -- \
  --mode train \
  --matrix "$MATRIX" \
  --difficulty medium \
  --freeze-recurrent \
  --epochs 30 \
  --patience 8 \
  --batch-size 64 \
  --log-every-seconds 30

python scripts/summarize_associative_sweep.py "$OUT"
cat "$OUT/leaderboard.csv"
cat "$OUT/paired_comparisons.csv"
```

For optic-flow sweeps, positive `delta_vs_*` and positive paired
`mean_delta` values mean the model has lower RMSE than the baseline.
